
Техническая реализация приложения

Наша задача — быстро запустить MVP мобильного AI-приложения для подготовки к экзаменам, и при этом заложить возможность масштабирования.

Главный принцип выбора стека: быстро → недорого → масштабируемо → удобно развивать.


---

1. **Платформа: мобильное приложение**

Выбираем: Flutter

Flutter позволяет писать один код сразу для iOS и Android.

Плюсы:
- быстрее разработка (не нужно две команды);
- дешевле MVP;
- одинаковый интерфейс на обеих платформах;
- высокая производительность (почти native);
- хорош для красивых UI.

---

Альтернатива: React Native

Плюсы:
- огромная экосистема;
- проще найти разработчиков.


Минусы:
- иногда слабее производительность;
- больше проблем с native-модулями.

---

Почему не native (Swift + Kotlin)?

Потому что:
- дорого;
- долго;
- нужно две команды.

Для MVP это неэффективно.

---

Вывод:

Flutter — лучший выбор для быстрого запуска.


---

2. **Frontend архитектура**

**Подход: Clean Architecture** (послойная разработка)

Почему:
- код легко поддерживать;
- удобно тестировать;
- проще масштабировать.

Структура:
	UI Layer
	↓
	State Management
	↓
	Business Logic
	↓
	Data Layer
	↓
	API / Database


---

**State Management**

Выбираем: Riverpod

Он позволяет централизованно хранить данные пользователя, результаты тестов и прогресс, автоматически обновляя интерфейс при изменениях. Это делает приложение стабильным и удобным для масштабирования.

Плюсы:
- современный
- безопаснее Provider
- хорошо масштабируется

Альтернатива:
Bloc — мощный, но сложнее;

Provider — проще, но хуже для больших проектов.

---

3. **Backend**

Выбираем: Firebase

Почему:
- не нужно писать backend с нуля;
- готовая авторизация;
- база данных;
- push-уведомления;
- аналитика;
- быстрое MVP.

Компоненты:
- Firebase Authentication
- Cloud Firestore
- Firebase Cloud Messaging
- Firebase Analytics



---

Альтернатива:

Node.js + PostgreSQL

Когда лучше:
- если нужен сложный backend;
- своя логика;
- высокая кастомизация.

Почему не сейчас:
- дольше запуск.



---

Вывод:

Для MVP — Firebase, для масштабирования позже можно перейти на собственный backend.


---

4. **База данных**

Выбираем: Cloud Firestore

Что храним:
- профиль ученика;
- результаты тестов;
- слабые темы;
- прогресс;
- учебный план.

Почему:
- realtime
- удобно под мобильные приложения;
- гибкая схема.

---

Почему не SQL?

SQL хорош для сложной аналитики, но MVP проще на NoSQL.


---

5. **AI-компонент (главное)**

Выбираем: OpenAI API

Модель: GPT

---

**Что делает AI?**

1. Объясняет ошибки

покажет где ошибка, почему, как решать правильно

---

2. **Строит персональный план**

На основе:
- результатов теста;
- слабых тем;
- целевого балла.

Пример: «У тебя слабая тригонометрия → 5 дней на повторение».

---

3. **AI-репетитор**

Чат 24/7:
- задаёт уточняющие вопросы;
- помогает шаг за шагом.

---

Почему OpenAI?

Плюсы:
- лучший reasoning;
- сильное объяснение;
- API готов сразу

---

Альтернатива: Yandex GPT

Плюсы:
- локальный рынок;
- русский язык.
Минусы:
- пока слабее reasoning.

---

В будущем:

Можно обучить собственную модель на заданиях ЕГЭ.


---

6. **AI-архитектура**

Используем RAG (Retrieval-Augmented Generation)

Схема:

Запрос ученика
↓
Поиск в базе решений ЕГЭ
↓
Передача контекста в GPT
↓
Ответ

выше точность;
ответы ближе к школьной программе.

---

7. **Хранение учебного контента**

Выбираем: Cloud Storage

Храним:
- pdf;
- изображения;
- видео;
- задания.

---

8. **Push-уведомления**
Через: Firebase Cloud Messaging

Примеры:
- «Пора решить 5 задач»
- «Ты не заходил 2 дня»
- Зачем: повышение retention.

---

9. **Аналитика**

Выбираем: Firebase Analytics + Mixpanel (позже)

Метрики:
DAU;
retention;
completion rate;
рост среднего балла.

---

10. **Безопасность**
	1. авторизация через email/google;
	2. шифрование данных;
	3. минимизация персональных данных;
	4. соответствие GDPR-подходу.
	5. Так как пользователи — дети, это важно.
